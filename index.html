저는 사용자의 아이디어를 기술로 구현하는 과정에 매력을 느끼는 개발자입니다. 특히 React와 Python(FastAPI)을 함께 사용하여 풀스택으로 서비스를 구축하는 경험을 즐깁니다.

이번 '플랜-AI' 프로젝트는 저에게 예상치 못한 큰 도전이었습니다. 생성형 AI(Gemini)와 네이버 지도 API를 연동하는 과정에서, 제가 사용한 NCP(Naver Cloud Platform) 계정이 'VPC' 전용 환경이라 프론트엔드에서 Maps JavaScript API (Web JS)의 인증에 계속 실패하는 기술적 한계에 부딪혔습니다. '동적 지도'라는 핵심 기능을 띄울 수 없는 막다른 상황이었습니다.

저는 포기하는 대신, Static Map (REST API)이라는 대안을 찾아냈습니다. 백엔드에서 AI의 응답을 기반으로 지도 '이미지'를 직접 생성하고, 이를 Base64로 인코딩하여 프론트엔드로 전달하는 방식으로 아키텍처를 전면 수정하여 문제를 해결했습니다. 이 경험을 통해 "길이 막혔을 때 포기하는 것이 아니라, 다른 길을 찾아내는 것"이 개발자의 역량임을 배울 수 있었습니다.

프로젝트: 플랜-AI (Plan-AI)

문제 정의

사용자에게 AI 기반 여행 추천 코스를 '시각적'으로 제공하는 것이 목표였습니다. 하지만 개발에 사용한 NCP 계정이 'VPC' 전용 환경이라, 프론트엔드(React)에서 Maps JavaScript API (Web JS)의 인증(Auth)에 지속적으로 실패하는 기술적 한계에 부딪혔습니다. (브라우저 LatLngBounds 에러 발생) 이 문제를 해결하고 사용자에게 '지도'라는 핵심 시각화 기능을 제공하는 것이 당면 과제였습니다.

아키텍처/설계

React(프론트엔드)와 Python(FastAPI)(백엔드)를 분리한 REST API 아키텍처를 채택했습니다.
핵심 설계 변경: 프론트엔드의 Dynamic Map 인증 문제를 해결하기 위해, **Static Map (REST API)**으로 아키텍처를 '피벗(Pivot)'했습니다. 모든 API 호출은 Python 백엔드가 전담하며, '지도 생성'까지 백엔드에서 처리하도록 설계했습니다. 백엔드는 AI가 생성한 장소 목록을 Geocoding으로 좌표 변환 후, Static Map API를 호출하여 마커와 경로가 포함된 지도 '이미지'를 Base64로 생성해 React(<img> 태그)로 전달합니다.

문제 해결 로드맵 (1인 프로젝트)

단순한 기능 구현을 넘어, 1인 개발자로서 3가지 핵심 기술적 난제를 진단하고 해결했습니다.

STEP 1
문제: Google AI `v1beta` 404 에러

API 호출 시, `gemini-1.0-pro` 모델을 찾을 수 없다는 고질적인 `v1beta` 에러가 발생했습니다.

▶ 해결: **Python 3.13**의 호환성 문제로 규명, `py -3.11` 및 `venv` 환경을 재설정하여 해결했습니다.

문제: `gemini-pro` 무료 등급 호출 실패

환경 재설정 후에도 유료 모델 API 호출이 실패했습니다.

▶ 해결: **`gemini-2.5-flash` (최신 무료 등급)** 모델을 사용하도록 디버깅하여 API 호출을 최종적으로 성공시켰습니다.

STEP 3
문제: `Dynamic Map` (VPC) 인증 실패

백엔드(AI) 성공 후, 프론트엔드에서 `LatLngBounds` 에러가 발생하며 지도를 띄울 수 없었습니다.

▶ 해결: 'VPC' 계정은 'Web JS' API를 쓸 수 없음을 규명, **`Static Map (Base64)` 방식으로 아키텍처를 '피벗'**하여 프로젝트를 완성시켰습니다.

핵심 기능 상세

Gemini (`gemini-2.5-flash`) 기반 코스 생성
  • Google Gemini API (`gemini-2.5-flash`)를 활용하여 사용자의 (여행지, 기간, 테마) 입력에 맞는 여행 장소 목록을 생성합니다.
  • AI가 `JSON List` 형식으로만 응답하도록 강력한 프롬프트 엔지니어링을 적용했습니다.
  • `json.loads`를 통해 응답을 파싱하여 백엔드에서 안정적으로 데이터를 처리합니다.
NCP 'VPC/REST API' 환경 지도 시각화
  • `Dynamic Map (Web JS)` 인증 실패 문제를 **Static Map (REST API)**으로 해결했습니다.
  • `Geocoding API`로 AI가 제안한 장소명을 위도/경도 좌표로 변환합니다.
  • 획득한 좌표 목록을 `Static Map API` 명세에 맞춰 마커(markers)와 경로(path) 파라미터로 조합합니다.
  • Python `requests`와 인증 헤더를 사용해 `Static Map API`를 호출, 지도 '이미지'를 응답받습니다.
  • 응답받은 이미지를 `Base64`로 인코딩하여, 프론트엔드 <img> 태그가 즉시 사용할 수 있는 텍스트로 변환하여 전달합니다.

결과 및 성과

  • 'VPC'라는 제한된 NCP 개발 환경의 한계를 `REST API` 우회 전략으로 극복하고 AI 여행 플래너의 핵심 기능(경로 시각화)을 '완성'.
  • "VErcel/Netlify 배포 완료 및 동료 10명 대상 정량/피드백 수집 완료."

회고 (Lessons Learned)

배운 점 (What I Learned)
  • API 명세서의 '제공 방식'(REST vs. Web JS)과 '인증 환경'(VPC vs. Public)을 확인하는 것이 얼마나 중요한지 뼈저리게 배웠습니다.
  • Python 버전 호환성(`v1beta` 에러) 및 API 모델 등급(`gemini-2.5-flash`)이 실제 API 호출에 미치는 영향을 직접 디버깅하며 `venv`의 중요성을 깨달았습니다.
  • 초기 기획(동적 지도)이 막혔을 때 포기하지 않고, `Static Map`이라는 대안 아키텍처로 '피벗'하는 문제 해결 과정을 경험했습니다.
개선점 (What's Next)
  • 향후 NCP '공개(Public)' 포털 계정을 발급받아, `Dynamic Map (Web JS)`을 활용한 원본 기획(동적 지도)으로 고도화하고 싶습니다.
  • 백엔드에서 매번 생성하는 `Base64` 이미지를 S3/R2 같은 클라우드 스토리지에 캐싱하여 성능을 최적화하고 싶습니다.
